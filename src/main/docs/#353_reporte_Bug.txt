BUG #353 - HISTORIAL DE ESTADOS DE ENVÍO INCORRECTO Y RUTEO RESILIENTE
═══════════════════════════════════════════════════════════════════════════════

Integrante a cargo: Jonatan

OBJETIVO:
Solucionar el error en la tabla de auditoría donde los cambios de estado registraban el mismo valor para el estado anterior y el nuevo (ej: "De EN_TRANSITO a EN_TRANSITO"). Además, asegurar que el sistema no se bloquee (Error 500) si la API de mapas (GraphHopper) falla al calcular la ruta debido a coordenadas erróneas o caídas de red.

═══════════════════════════════════════════════════════════════════════════════

CLASE MODIFICADA: EnvioService.java
═══════════════════════════════════════════════════════════════════════════════

Ubicación: src/main/java/com/logitrack/sistema_logistica/service/EnvioService.java

QUE HABÍA ANTES:

El método actualizarEstadoOperativo guardaba la entidad en la base de datos antes de delegar la creación del historial, provocando que Hibernate leyera el nuevo estado como si fuera el anterior.

El llamado a trackingService.generarYGuardarRuta rompía el flujo de actualización de estado si las coordenadas no eran encontradas por GraphHopper.

CÓDIGO COMPLETAMENTE REESCRITO Y MODIFICADO:

A. Bloque de actualizarEstadoOperativo:

Java
@Transactional
public Envio actualizarEstadoOperativo(String idEnvio, EnvioOperativoDTO dto, Authentication auth) {
        Envio envioExistente = envioRepository.findById(idEnvio)
                        .orElseThrow(() -> new RuntimeException("No se encontró el envío con ID: " + idEnvio));

        EstadoEnvio estadoAnterior = envioExistente.getEstadoActual();
        boolean estadoCambiado = (dto.getEstado() != null && dto.getEstado() != estadoAnterior);
        String prioridadFinal = envioExistente.getPrioridadIa();
        boolean prioridadCambiada = false;

        // 1. Validación de Prioridad
        if (dto.getPrioridadIa() != null && !dto.getPrioridadIa().equals(envioExistente.getPrioridadIa())) {
                boolean esSupervisor = auth.getAuthorities().stream()
                                .anyMatch(a -> a.getAuthority().equals("ROLE_SUPERVISOR"));

                if (!esSupervisor) {
                        throw new RuntimeException("La prioridad del envío solo puede ser modificada por un supervisor.");
                }
                prioridadFinal = dto.getPrioridadIa();
                prioridadCambiada = true;
        }

        Usuario usuarioModificador = usuarioRepository.findByUsername(auth.getName())
                        .orElseThrow(() -> new RuntimeException("Usuario no encontrado"));

        // 2. Si el estado cambió, delegamos TODO al método centralizado
        if (estadoCambiado) {
                return actualizarEstadoYPrioridad(
                                idEnvio,
                                dto.getEstado().name(),
                                prioridadFinal,
                                usuarioModificador,
                                TipoEvento.CAMBIO_ESTADO);
        }

        // 3. Si NO cambió el estado, pero SÍ la prioridad
        if (prioridadCambiada) {
                envioExistente.setPrioridadIa(prioridadFinal);
                Envio envioGuardado = envioRepository.save(envioExistente);
                auditoriaService.registrarEvento(envioGuardado, usuarioModificador, TipoEvento.CAMBIO_PRIORIDAD, estadoAnterior, estadoAnterior);
                return envioGuardado;
        }

        return envioExistente;
}

EXPLICACIÓN:

EstadoEnvio estadoAnterior = envioExistente.getEstadoActual();: Congela y almacena en la memoria el estado original (ej. PENDIENTE) apenas arranca el método, antes de que Hibernate actualice el objeto.

Se eliminó un envioRepository.save() prematuro para que la persistencia se realice únicamente en el método centralizado y luego de validar qué acción corresponde.

try-catch en Ruteo: Encapsula la llamada a trackingService para que, si GraphHopper lanza una excepción por coordenadas mal formadas (ej: sin el signo menos), el programa atrape el error en consola y continúe el flujo guardando el estado sin interrumpir la operación del usuario.

POR QUE:

Soluciona el problema de los historiales duplicados (Bug #353) al asegurar que se toma una "foto" del estado viejo antes de efectuar cualquier sobrescritura en la base de datos.

Aporta resiliencia al sistema. La logística operativa (cambiar a EN_TRANSITO) es más crítica que el dibujo del mapa. Si el mapa falla, el camión igual debe poder despacharse administrativamente.

FLUJO COMPLETO DE EJECUCIÓN
═══════════════════════════════════════════════════════════════════════════════

Usuario llama:
PATCH http://localhost:8080/api/envios/{id}/operativo

EnvioController.actualizarOperativaEnvio() recibe el JSON con el nuevo estado.

Llama a envioService.actualizarEstadoOperativo().

El servicio busca el envío y clona el estado original (estadoAnterior = PENDIENTE).

Si hay cambio de estado, delega la entidad, el estado nuevo y el estado viejo clonado a actualizarEstadoYPrioridad().

Se actualiza el objeto en memoria a EN_TRANSITO.

Se intenta generar la ruta en GraphHopper. Si falla (por coordenadas erróneas), se registra el error en consola y sigue adelante.

Se guarda el envío actualizado (envioRepository.save()).

Se invoca a auditoriaService.registrarEvento(), pasándole el estado viejo congelado y el estado nuevo actualizado.

Spring serializa y devuelve el envío en JSON al frontend con un Status 200 OK.

CÓMO PROBAR
═══════════════════════════════════════════════════════════════════════════════

Asegurarse de que el servidor está corriendo y obtener un Token JWT de Supervisor.

Buscar un envío en estado PENDIENTE.

En Thunder Client, enviar un PATCH a: http://localhost:8080/api/envios/LT-XXXXXX/operativo
Body:

JSON
{
  "estado": "EN_TRANSITO",
  "prioridadIa": "ALTA"
}
El endpoint debe responder 200 OK (a pesar de que GraphHopper pueda advertir un error en la consola de Spring).

En Thunder Client, realizar un GET a http://localhost:8080/api/envios/LT-XXXXXX/historial

Verificar que el objeto más reciente en el JSON devuelva correctamente: "estadoAnterior": "PENDIENTE" y "estadoNuevo": "EN_TRANSITO".
