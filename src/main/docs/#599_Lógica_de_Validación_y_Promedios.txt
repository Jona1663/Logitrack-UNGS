TAREA #599 - LÓGICA DE VALIDACIÓN Y PROMEDIOS EN EVALUACIÓN DE FATIGA
═══════════════════════════════════════════════════════════════════════════════

Integrante a cargo: Jonatan

OBJETIVO:
Implementar en `EvaluacionFatigaService` la lógica transaccional para procesar el test de fatiga.
Rechazar automáticamente tiempos de reacción fuera del rango permitido y notificar el rechazo por WebSocket usando la configuración existente (Stomp/SockJS).

═══════════════════════════════════════════════════════════════════════════════
1. SERVICIO MODIFICADO: EvaluacionFatigaService.java
═══════════════════════════════════════════════════════════════════════════════

Ubicación: src/main/java/com/logitrack/sistema_logistica/service/EvaluacionFatigaService.java

Propósito:
- Implementar el método transaccional que recibe el test de fatiga.
- Aplicar reglas de validación de tiempos de reacción.
- Emitir notificación en caso de rechazo por tiempo inválido.

Qué se hizo:
- Se definió un método público transaccional en `EvaluacionFatigaService`.
- El método recibe el DTO del test o los parámetros requeridos.
- Se validó el tiempo de reacción en milisegundos.
- Se rechazó automáticamente el test cuando el valor era:
  - menor a 100 ms → posible trampa o bot
  - mayor a 600 ms → fatiga extrema
- Se dejó claro que estos umbrales son configurables y pueden cambiarse.

Explicación:
- `@Transactional` asegura que la operación y la notificación se ejecutan dentro de la misma transacción.
- Si el test es rechazado, no se persiste como aprobado y se dispara la alerta por WebSocket.
- El servicio mantiene la lógica de negocio centralizada.

═══════════════════════════════════════════════════════════════════════════════
2. LÓGICA DE VALIDACIÓN IMPLEMENTADA
═══════════════════════════════════════════════════════════════════════════════

Reglas de validación:
- `milisegundos < 100` → rechazo inmediato (trampa/bot).
- `milisegundos > 600` → rechazo inmediato (fatiga extrema).

Resultados:
- Si el tiempo está dentro del rango permitido, el test se considera válido.
- Si el tiempo está fuera del rango, se marca como rechazado.

Motivo:
- Garantizar calidad de los datos recibidos.
- Evitar pruebas inválidas debido a entradas no reales.
- Detectar tanto respuestas demasiado rápidas como demasiado lentas.

═══════════════════════════════════════════════════════════════════════════════
3. NOTIFICACIÓN POR WEBSOCKET
═══════════════════════════════════════════════════════════════════════════════

Ubicación: Configuración existente de WebSocket STOMP/SockJS en el proyecto.

Qué se hizo:
- Se configuró el servicio para emitir un mensaje cuando el test es rechazado.
- La notificación utiliza el canal WebSocket ya disponible.
- Se aprovechó la infraestructura de STOMP/SockJS existente sin crear un nuevo mecanismo.

Explicación:
- Al detectar un rechazo, el servicio publica un evento de WebSocket.
- Esto permite que el cliente o dashboard reciba la alerta en tiempo real.
- El mensaje puede incluir el estado de rechazo y el motivo aplicado.

═══════════════════════════════════════════════════════════════════════════════
4. FLUJO COMPLETO DE LA TAREA
═══════════════════════════════════════════════════════════════════════════════

1. El cliente envía el test de fatiga al servicio.
2. `EvaluacionFatigaService` inicia la transacción.
3. Se lee el valor de `milisegundos`.
4. Se evalúa si está fuera del rango permitido.
5. Si está fuera del rango:
   - Se rechaza el test.
   - Se emite un mensaje de WebSocket usando STOMP/SockJS.
6. Si está dentro del rango:
   - El test continúa como válido.
