2. CLASES MODIFICADAS: Repositorios (Capa de Datos)
═══════════════════════════════════════════════════════════════════════════════

Ubicación:

src/main/java/com/logitrack/sistema_logistica/repository/IncidenciaRepository.java

src/main/java/com/logitrack/sistema_logistica/repository/EnvioRepository.java

QUE SE AGREGÓ / MODIFICÓ:
Se agregaron consultas optimizadas para contar registros directamente en la base de datos usando Spring Data JPA.

CÓDIGO:

Java
// En IncidenciaRepository.java
long countByFechaReporteBetween(LocalDateTime inicio, LocalDateTime fin);

// En EnvioRepository.java (Se reemplazó el método viejo por estos dos)
@Query("SELECT COUNT(e) FROM Envio e WHERE e.fechaLlegada IS NOT NULL AND e.fechaLlegada <= e.fechaEstimadaLlegada AND e.fechaCreacion BETWEEN :inicio AND :fin")
long countEnviosATiempoEntreFechas(@Param("inicio") LocalDateTime inicio, @Param("fin") LocalDateTime fin);

@Query("SELECT COALESCE(SUM(e.kgOrigen), 0L) FROM Envio e WHERE e.fechaLlegada IS NOT NULL AND e.fechaLlegada <= e.fechaEstimadaLlegada AND e.fechaCreacion BETWEEN :inicio AND :fin")
long sumKilosATiempoEntreFechas(@Param("inicio") LocalDateTime inicio, @Param("fin") LocalDateTime fin);
EXPLICACIÓN:

En IncidenciaRepository, Spring Boot genera automáticamente la query SQL solo con leer el nombre del método (countBy...Between).

En EnvioRepository, separamos el conteo de viajes y la suma de kilos en dos consultas nativas rápidas.

POR QUE:

Es crucial para el rendimiento. Traer todos los objetos a Java para contarlos colapsaría la memoria (RAM) del servidor cuando LogiTrack tenga miles de viajes. Hacer el COUNT y SUM en PostgreSQL devuelve un simple número de forma casi instantánea.

3. CLASE MODIFICADA: ReporteService.java (Reglas de Negocio)
═══════════════════════════════════════════════════════════════════════════════

Ubicación: src/main/java/com/logitrack/sistema_logistica/service/ReporteService.java

QUE SE AGREGÓ:
Se inyectó el repositorio de incidencias y se implementó la matemática segura para el cálculo de porcentajes.

CÓDIGO CLAVE AGREGADO:

Java
// En obtenerReporte():
long totalIncidencias = incidenciaRepository.countByFechaReporteBetween(inicio, fin);

// En obtenerMetricasATiempo():
long viajesATiempo = envioRepository.countEnviosATiempoEntreFechas(inicio, fin);
long totalCompletados = envioRepository.countCompletadosEntreFechas(inicio, fin);

double porcentaje = 0.0;
if (totalCompletados > 0) { // <-- PREVENCIÓN DE ERROR CRÍTICO
    porcentaje = Math.round(((double) viajesATiempo / totalCompletados) * 100.0);
}
EXPLICACIÓN:

Se agrupan los resultados de las queries de los repositorios.

Se hace un cast a (double) para que la división no pierda los decimales.

Validación Empty State: El bloque if (totalCompletados > 0) verifica que haya viajes antes de dividir.

POR QUE:

Si el supervisor filtra un mes sin actividad y el código intenta dividir por cero, el servidor arrojaría una excepción ArithmeticException (Error 500) y rompería la interfaz. Con esta validación, si no hay datos, simplemente devuelve 0.0.

5. CLASE MODIFICADA: ReporteController.java (Manejo de HTTP 204)
═══════════════════════════════════════════════════════════════════════════════

Ubicación: src/main/java/com/logitrack/sistema_logistica/controller/ReporteController.java

QUE SE HIZO:
Refactorización para capturar la señal de "Empty State" y responder 204.

CÓDIGO MODIFICADO:

Java
@GetMapping("/operativo")
public ResponseEntity<?> reporteOperativo(...) {
    try {
        ReporteSimpleDTO reporte = reporteService.obtenerReporte(fechaInicio, fechaFin);
        return ResponseEntity.ok(reporte);
    } catch (RuntimeException e) {
        if ("EMPTY_STATE".equals(e.getMessage())) {
            // El frontend recibe un 204 y activa el mensaje "No hay datos"
            return ResponseEntity.noContent().build(); 
        }
        return ResponseEntity.badRequest().body(e.getMessage());
    }
}
EXPLICACIÓN:

Cambiamos el retorno a ResponseEntity<?> (comodín) para poder devolver datos o una respuesta vacía.

Capturamos la excepción "EMPTY_STATE" y devolvemos 204 No Content.

POR QUE:

El frontend no debe romper al recibir un JSON vacío o ceros sin sentido; al recibir un 204, puede ocultar los gráficos y mostrar un mensaje amigable al usuario.

FLUJO COMPLETO DE EJECUCIÓN (Ejemplo: Eficiencia)
═══════════════════════════════════════════════════════════════════════════════

El frontend (Dashboard) llama:
GET http://localhost:8080/api/reportes/a-tiempo?fechaInicio=2026-06-01&fechaFin=2026-06-30

ReporteController recibe la petición, extrae las fechas y llama a reporteService.obtenerMetricasATiempo().

El servicio solicita a EnvioRepository tres datos:

Cuántos viajes llegaron a tiempo en junio.

Cuántos kilos suman esos viajes.

Cuántos viajes se completaron en total en junio.

El servicio calcula el porcentaje de forma segura.

Construye el ReporteEficienciaDTO con los tres valores y se lo devuelve al controlador.

El controlador responde con un 200 OK y Spring convierte el DTO a este JSON:
{
"viajesATiempo": 5,
"kilosATiempo": 140000,
"porcentajeATiempo": 50.0
}
