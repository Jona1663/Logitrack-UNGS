TAREA #232 - ENDPOINT DE REPORTE OPERATIVO (Viajes y Kilos)
═══════════════════════════════════════════════════════════════════════════════

OBJETIVO:
Crear un endpoint REST que devuelva dos métricas simples:
1. Cantidad total de viajes (envíos) registrados
2. Cantidad total de kilos transportados
Ideal para exportar directamente a Excel sin complejidades de fecha.

═══════════════════════════════════════════════════════════════════════════════
1. CLASE CREADA: ReporteSimpleDTO.java
═══════════════════════════════════════════════════════════════════════════════

Ubicación: src/main/java/com/logitrack/sistema_logistica/dto/ReporteSimpleDTO.java

Propósito: 
- Transportar datos de la capa de servicio al controlador
- Ser serializado automáticamente a JSON por Spring
- Contener SOLO lo necesario: totalViajes y totalKilos

Código:
```java
@Data
@NoArgsConstructor
@AllArgsConstructor
@Builder
public class ReporteSimpleDTO {
    private long totalViajes;
    private long totalKilos;
}
```

Explicación:
- @Data: Lombok genera getters, setters, equals, hashCode, toString
- @NoArgsConstructor: Constructor sin argumentos (requerido por Spring)
- @AllArgsConstructor: Constructor con todos los argumentos
- @Builder: Patrón Builder para crear instancias
- totalViajes: Contará la cantidad de registros en la tabla envios
- totalKilos: Sumará kg_destino (o kg_origen si es NULL)

═══════════════════════════════════════════════════════════════════════════════
2. CLASE MODIFICADA: EnvioRepository.java
═══════════════════════════════════════════════════════════════════════════════

Ubicación: src/main/java/com/logitrack/sistema_logistica/repository/EnvioRepository.java

QUE SE AGREGÓ:
Se agregó un método de query nativa para sumar kilos de forma eficiente.

CÓDIGO AGREGADO AL FINAL:
```java
@Query(value = "SELECT COALESCE(SUM(COALESCE(kg_destino, kg_origen)), 0) FROM envios", nativeQuery = true)
Long sumKilos();
```

EXPLICACIÓN:
- @Query(nativeQuery=true): Usa SQL puro en lugar de JPQL
- COALESCE(kg_destino, kg_origen): Si kg_destino es NULL, usa kg_origen
- COALESCE(..., 0): Si ambos son NULL, devuelve 0
- Long (en lugar de long): Permite devolver NULL si no hay datos
- Esta query se ejecuta directamente en la base de datos, es muy eficiente

POR QUE:
- Es más rápido que iterar en Java
- Maneja correctamente los NULLs directamente en la BD
- Devuelve un número listo para usar en el DTO

═══════════════════════════════════════════════════════════════════════════════
3. CLASE MODIFICADA: Envio.java
═══════════════════════════════════════════════════════════════════════════════

Ubicación: src/main/java/com/logitrack/sistema_logistica/model/Envio.java

QUE SE AGREGÓ:
Se agregó un import y una anotación en el campo fechaCreacion.

CÓDIGO MODIFICADO:
Antes:
```java
import jakarta.persistence.*;
import lombok.Data;

@Column(name = "fecha_creacion", updatable = false)
private LocalDateTime fechaCreacion;
```

Después:
```java
import jakarta.persistence.*;
import org.hibernate.annotations.CreationTimestamp;
import lombok.Data;

@CreationTimestamp
@Column(name = "fecha_creacion", updatable = false)
private LocalDateTime fechaCreacion;
```

QUE CAMBIÓ:
- Se importó: org.hibernate.annotations.CreationTimestamp
- Se agregó: @CreationTimestamp antes del campo fechaCreacion

EXPLICACIÓN:
@CreationTimestamp es una anotación de Hibernate que:
- Asigna automáticamente la fecha/hora actual cuando se crea un registro
- Se ejecuta ANTES de persistir en la BD
- No permite null en este campo
- Hace que cada envío siempre tenga una fecha de creación válida

POR QUE:
- Antes: fechaCreacion podía ser NULL
- Si es NULL: La query SUM() del reporte no encuentra registros
- Ahora: Todos los envíos tienen fechaCreacion asignada automáticamente
- Resultado: El reporte siempre devuelve datos correctos

4. CLASE MODIFICADA: ReporteService.java
═══════════════════════════════════════════════════════════════════════════════

Ubicación: src/main/java/com/logitrack/sistema_logistica/service/ReporteService.java

QUE HABÍA ANTES:
- Contenía múltiples métodos complejos con queries nativas
- Métodos para filtrar por fecha (obtenerTotales, obtenerTotalesPorEstadoEntregado, etc)
- DTOs complejos (ReporteTotalesDTO, ReporteEstadoDTO)
- EntityManager para ejecutar queries manualmente

CÓDIGO COMPLETAMENTE REESCRITO:
```java
@Service
public class ReporteService {

    @Autowired
    private EnvioRepository envioRepository;

    @Transactional(readOnly = true)
    public ReporteSimpleDTO obtenerReporte() {
        long totalViajes = envioRepository.count();  // SELECT COUNT(*) FROM envios
        Long totalKilos = envioRepository.sumKilos();
        
        if (totalKilos == null) {
            totalKilos = 0L;
        }

        return ReporteSimpleDTO.builder()
                .totalViajes(totalViajes)
                .totalKilos(totalKilos)
                .build();
    }
}
```

EXPLICACIÓN LÍNEA POR LÍNEA:

1. @Autowired private EnvioRepository envioRepository;
   - Inyecta el repositorio para acceder a la BD
   
2. @Transactional(readOnly = true)
   - Abre una transacción de solo lectura (más eficiente)
   
3. long totalViajes = envioRepository.count();
   - Devuelve la cantidad de filas en la tabla envios
   - Spring Data genera automáticamente: SELECT COUNT(*) FROM envios
   
4. Long totalKilos = envioRepository.sumKilos();
   - Llama a la query nativa que creamos en el repositorio
   - Devuelve la suma de kilos (con COALESCE para NULLs)
   
5. if (totalKilos == null) { totalKilos = 0L; }
   - Si no hay envíos, totalKilos podría ser NULL
   - Lo convertimos a 0 para que el JSON sea válido
   
6. return ReporteSimpleDTO.builder()...build();
   - Crea y devuelve un DTO con los dos valores

POR QUE:
- Simplicidad: Un solo método público que hace lo necesario
- Eficiencia: Las queries se ejecutan en la BD, no en Java
- Mantenibilidad: Fácil de entender y modificar

5. CLASE MODIFICADA: ReporteController.java
═══════════════════════════════════════════════════════════════════════════════

Ubicación: src/main/java/com/logitrack/sistema_logistica/controller/ReporteController.java

QUE HABÍA ANTES:
- Múltiples endpoints (@GetMapping)
- Parámetros complejos (@RequestParam de fechas)
- Lógica para resolver presets (hoy, ultimos7d)
- Devolvía diferentes DTOs según el endpoint

CÓDIGO COMPLETAMENTE REESCRITO:
```java
@RestController
@RequestMapping("/api/reportes")
public class ReporteController {

    @Autowired
    private ReporteService reporteService;

    @GetMapping("/operativo")
    public ResponseEntity<ReporteSimpleDTO> reporteOperativo() {
        ReporteSimpleDTO reporte = reporteService.obtenerReporte();
        return ResponseEntity.ok(reporte);
    }
}
```

EXPLICACIÓN:

1. @RestController
   - Marca la clase como controlador REST
   - Todos los métodos devuelven JSON automáticamente
   
2. @RequestMapping("/api/reportes")
   - Prefijo de URL para todos los endpoints de esta clase
   
3. @GetMapping("/operativo")
   - Endpoint: GET /api/reportes/operativo
   - Sin parámetros adicionales
   
4. public ResponseEntity<ReporteSimpleDTO> reporteOperativo()
   - ResponseEntity permite controlar status HTTP (200, 404, 500, etc)
   - Genérico <ReporteSimpleDTO> especifica qué tipo se serializa
   
5. reporteService.obtenerReporte()
   - Llama al servicio para obtener los datos
   
6. return ResponseEntity.ok(reporte);
   - Devuelve status 200 OK + JSON con los datos

EJEMPLO DE RESPUESTA:
```json
{
  "totalViajes": 10,
  "totalKilos": 245000
}
```

POR QUE:
- Sin fechas: Se elimina la complejidad de parámetros
- Sin filtros: Devuelve el total global del sistema
- Un endpoint: Más fácil de usar y mantener
- Listo para Excel: JSON simple sin datos anidados

FLUJO COMPLETO DE EJECUCIÓN
═══════════════════════════════════════════════════════════════════════════════

Usuario llama:
GET http://localhost:8080/api/reportes/operativo

1. ReporteController.reporteOperativo() recibe la request
   
2. Llama: reporteService.obtenerReporte()
   
3. ReporteService.obtenerReporte() hace:
   - envioRepository.count() 
     → SQL: SELECT COUNT(*) FROM envios
     → Devuelve: 10
   
   - envioRepository.sumKilos()
     → SQL: SELECT COALESCE(SUM(COALESCE(kg_destino, kg_origen)), 0) FROM envios
     → Devuelve: 245000
   
   - Construye ReporteSimpleDTO(10, 245000)
   
4. ReporteController devuelve ResponseEntity.ok(reporte)
   
5. Spring serializa el DTO a JSON:
   {
     "totalViajes": 10,
     "totalKilos": 245000
   }
